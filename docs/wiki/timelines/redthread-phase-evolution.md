# RedThread Phase Evolution

## Timeline

### Phase 1 — Foundation and PAIR
RedThread established the baseline engine, CLI, judge role, and first attack algorithm.

### Phase 2 — PAIR refinement
The project hardened PAIR and added persona-based diversity.

### Phase 3 — TAP
RedThread added tree-based attack search and pruning.

### Phase 4 — LangGraph orchestration
The project shifted from single-algorithm execution toward coordinated multi-agent workflow.

### Phase 4.5 — Defense synthesis and telemetry
The attack loop became self-healing through isolate → classify → generate → validate → deploy behavior, while telemetry introduced utility-aware monitoring.

### Phase 5A–5D — Reliability and operations hardening
The project added anti-hallucination standards, ARIMA/ASI, continuous monitoring, and CI/CD gates.

### Phase 6A — Crescendo
Multi-turn escalation became a first-class attack family.

### Phase 7A–7B — Bounded autoresearch era
The project direction shifted from "add more algorithms first" toward safe self-improvement through bounded offense and defense mutation lanes.

## Current state

The documented direction is now bounded self-improvement with explicit validation, replay evidence, and promotion discipline.

## Why this timeline matters

The project is no longer only an attack engine. It is increasingly a controlled self-improving system, which makes evidence, validation, and mutation boundaries central design concerns.
